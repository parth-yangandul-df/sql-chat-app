QueryWise Platform Architecture

A full-stack PRMS (Project Resource Management System) intelligence platform that routes natural-language questions through a LangGraph pipeline — using a 24-intent domain tool catalog for fast structured queries and a full semantic-layer LLM fallback for arbitrary questions.

Python 3.12 · FastAPI React 19 · TypeScript PostgreSQL · BigQuery · Databricks Anthropic · OpenAI · OpenRouter · Groq · Ollama LangGraph · Docker Compose

System Overview

The platform is organized into five logical layers. A user's natural language question flows top-down through presentation, API, semantic, LLM orchestration, and connector layers before results travel back up. The LLM orchestration layer is now a LangGraph StateGraph with two tracks: a fast domain tool path for classified PRMS intents, and a full LLM fallback path using the semantic layer for arbitrary questions.

Semantic Layer Core IP

The semantic layer is used by the LLM fallback path when the intent classifier confidence is below the threshold (0.65). It selects the most relevant database context for the NL question using a hybrid strategy: vector similarity + keyword matching + foreign key graph traversal. Imported business knowledge is chunked, embedded, and injected as additional context.

Context Building Pipeline (8 Steps)

1 Embed Question
Generate vector via OpenAI text-embedding-3-small (1536-dim) or Ollama nomic-embed-text (768-dim)
2 Vector Search
pgvector cosine distance against cached_tables.description_embedding — top 15 candidates
3 Keyword Matching
Extract keywords (stop-word filtered), ILIKE match against table/column names. Scores: exact=1.0, component=0.7, substring=0.5
4 FK Graph Expansion
From top 5 scored candidates, walk foreign key constraints to include necessary JOIN tables (relationship_score = 1.0)
5 Score & Prune
Weighted scoring formula, keep top 8 tables (MAX_CONTEXT_TABLES) with all their columns
6 Resolve Glossary, Metrics, Knowledge, Dictionary
Match business terms + metric names in question text & by embedding similarity. Find top 5 knowledge chunks by vector similarity (keyword ILIKE fallback). Fetch value mappings for selected columns.
7 Find Similar Sample Queries
Top 3 validated sample queries by embedding similarity — used as few-shot examples
8 Assemble Prompt
Format into structured sections: Schema → Relationships → Glossary → Metrics → Knowledge → Dictionary → Examples → Constraints

Relevance Scoring Formula

score = 0.5 × embedding + 0.3 × keyword + 0.2 × fk_proximity + glossary_boost

Semantic Metadata Types

Business Glossary

Maps business terms to SQL expressions. Matched by substring + embedding similarity.

Metric Definitions

Named calculations with aggregation types and dimensions. Matched by name + embedding.

Data Dictionary

Column value mappings so the LLM understands coded values.

Knowledge Base

Imported documentation (wikis, HTML pages) chunked and embedded for semantic retrieval. Top 5 chunks injected as === BUSINESS KNOWLEDGE ===

Sample Queries

Validated NL→SQL pairs used as few-shot examples. Top 3 by embedding similarity.

LLM Orchestration Layer

The core intelligence pipeline is a LangGraph StateGraph compiled as a singleton at startup. It implements two execution tracks: a fast domain tool track for classified PRMS intents (no SQL generation), and a full LLM fallback track with the semantic layer and 4 specialized agents.

LangGraph Graph Topology

classify_intent
     |
     ├─ confidence >= 0.65 → extract_params → run_domain_tool
     │                                         ├─ rows > 0 → interpret_result
     │                                         └─ 0 rows + fallback_intent? → run_fallback_intent
     │                                                          ├─ rows > 0 → interpret_result
     │                                                          └─ 0 rows   → llm_fallback
     └─ confidence < 0.65  → llm_fallback
                                    │
                              interpret_result
                                    │
                              write_history → END

GraphState — Shared Pipeline State

class GraphState(TypedDict):
    # Inputs
    question: str
    connection_id: str          # UUID as str
    session_id: str | None      # chat thread identifier
    conversation_history: list[dict]  # [{role, content}, ...]

    # Classification
    domain: str | None          # "resource"|"client"|"project"|"timesheet"
    intent: str | None          # e.g. "active_resources"
    confidence: float
    params: dict[str, Any]      # skill, dates, names (regex extracted)

    # Execution
    sql: str | None
    result: QueryResult | None
    generated_sql: str | None   # LLM path only
    retry_count: int

    # Interpretation
    answer: str | None
    highlights: list[str]
    suggested_followups: list[str]

    error: str | None

Graph Nodes

classify_intent

Method: Embeds the question, runs cosine similarity against all 24 pre-embedded catalog entries.

Output: domain, intent, confidence

Threshold: TOOL_CONFIDENCE_THRESHOLD=0.65 (env override). Logs top-3 scores for tuning.

extract_params

Method: Pure regex extraction — no LLM call. Extracts: skill, start_date, end_date, project_name, client_name, resource_name.

Patterns: ISO dates, Title Case names, project/client keywords.

run_domain_tool

Method: Dispatches to one of 4 domain modules (resource, client, project, timesheet) based on intent. Runs a pre-built parameterized SQL query directly against the connector.

Output: sql, result. If 0 rows and fallback_intent set, routes to run_fallback_intent.

llm_fallback

Trigger: Confidence < threshold, or domain tool 0-row result with no fallback.

Method: Full semantic context build → LLM SQL generation → validation → error-handler retry (max 3) → connector execution.

interpret_result

Input: Question + SQL + result rows

Output: answer (NL summary) + highlights + suggested_followups

Called for both paths.

write_history

Persists a QueryExecution record including session_id, SQL, timing, token counts, llm_provider, retry_count, and summary.

Complexity Router (LLM Fallback Path)

TierSignal KeywordsMax Tokens
Simple what is, show me, single entity lookups 512
Moderate group, total, average, join, top N, filter 768
Complex compare, trend, window, percentile, cohort, year-over-year 1,024
All tiers use the same model (configured via DEFAULT_LLM_PROVIDER / DEFAULT_LLM_MODEL). Only max_tokens differs. Temperature is always 0.0.

Four Specialized Agents (LLM Fallback)

QueryComposerAgent

Input: NL question + assembled context + conversation_history

Output: SQL + explanation + confidence (0–1) + tables_used + assumptions

Response: JSON parse with regex fallback for code-fenced SQL

SQLValidatorAgent

Level 1: Static regex blocklist (DROP, DELETE, INSERT, ALTER, CREATE, TRUNCATE, pg_sleep, etc.)

Level 2: Schema reference check via sqlparse (tables/columns exist)

Status: VALID | UNSAFE | SYNTAX_ERROR | SCHEMA_MISMATCH

ResultInterpreterAgent

Input: Question + SQL + results preview (max 20 rows)

Output: NL summary + key highlights + suggested follow-up questions

Also called for domain tool results.

ErrorHandlerAgent

Trigger: Validation failure OR database execution error

Input: Failed SQL + error + schema context + previous attempts

Output: Corrected SQL + explanation + should_retry flag

Max retries: 3

Query Flow Pipeline

End-to-end flow when a user submits a natural language question through POST /api/v1/query. The request carries an optional session_id and conversation_history for multi-turn chat context.

Domain Tool Path (Fast Track)

1 User submits NL question
"Show me active resources" — with optional session_id + conversation_history
2 classify_intent
Embed question → cosine similarity against 24 pre-embedded catalog entries → gets domain=resource, intent=active_resources, confidence=0.87
3 extract_params
Pure regex extraction. For this query: no skill/date/name params extracted. params={}
4 run_domain_tool
Dispatches to resource domain module → executes pre-built "active_resources" SQL with extracted params directly via connector. Returns rows.
5 interpret_result → write_history
LLM generates NL summary + highlights + follow-ups. Record saved to query_executions with llm_provider="domain_tool", session_id linked.

LLM Fallback Path (Arbitrary Questions)

1 User submits NL question
"What is the revenue trend by client over the last 6 months?" — confidence=0.41 < 0.65
2 llm_fallback — Context Builder
Embed question → hybrid table search (vector + keyword + FK) → resolve glossary/metrics/knowledge/dictionary → find sample queries → assemble structured prompt
3 LLM Router
Estimate complexity (complex: "trend", "over") → select configured provider/model with max_tokens=1024, temperature=0.0
4 QueryComposerAgent
LLM generates SQL from context + question + conversation_history. Parses JSON response (or fallback regex extraction). Returns SQL + explanation + confidence.
5 SQLValidatorAgent
Static blocklist check (DDL/DML/injection) + schema reference validation (sqlparse). If INVALID → ErrorHandlerAgent retries up to 3x.
Retry loop (max 3) — ErrorHandlerAgent corrects SQL
6 Execute via Connector
Read-only transaction + statement timeout + auto LIMIT wrapping. If DB error → ErrorHandlerAgent retries up to 3x.
7 interpret_result → write_history
LLM generates NL summary, highlights, follow-ups. Record saved with session_id, llm_provider, model, token counts, and retry_count.

Security Architecture

Defense-in-depth strategy with multiple independent safety layers, ensuring no single bypass can allow destructive operations.

🔒

Read-Only Enforcement

PostgreSQL: asyncpg transaction(readonly=True). BigQuery & Databricks: read-only enforced via SQL blocklist (no native read-only mode).

🚫

SQL Blocklist

Static regex patterns block DDL, DML, admin commands, injection patterns (pg_sleep, dblink), BigQuery ops (EXPORT DATA), and Databricks ops (COPY INTO, OPTIMIZE, VACUUM).

🔑

Encrypted Credentials

Connection strings encrypted at rest with Fernet (AES-128-CBC). Never returned in API responses — only has_connection_string: boolean.

⏱

Statement Timeout

PostgreSQL: SET LOCAL statement_timeout. BigQuery & Databricks: asyncio.wait_for(). Default 30s, configurable per connection.

📊

Row Limits

Auto-wrapping queries with LIMIT N+1 to cap result size. Default 1,000 rows, configurable per connection. Truncation flag in response.

🌐

CORS Restriction

FastAPI CORS middleware restricts origins. Default: http://localhost:5173 and http://localhost:5174 only.
