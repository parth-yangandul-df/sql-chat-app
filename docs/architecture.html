Saras Platform Architecture

A full-stack application that translates natural language questions into SQL queries using a semantic metadata layer (glossary, metrics, dictionary, knowledge base), multi-agent LLM orchestration, and a pluggable database connector system.

Python 3.12 · FastAPI React 19 · TypeScript PostgreSQL · BigQuery · Databricks Claude · OpenAI Docker Compose

System Overview

The platform is organized into five logical layers. A user's natural language question flows top-down through presentation, API, semantic, LLM, and connector layers before results travel back up.

Semantic Layer Core IP

The semantic layer is the product's key differentiator. It selects the most relevant database context for each natural language question using a hybrid strategy: vector similarity + keyword matching + foreign key graph traversal. Imported business knowledge (Confluence pages, wiki articles, policy documents) is chunked, embedded, and injected as additional context.

Context Building Pipeline (8 Steps)

1 Embed Question
Generate vector via OpenAI text-embedding-3-small (1536-dim) or Ollama nomic-embed-text (768-dim)
2 Vector Search
pgvector cosine distance against cached_tables.description_embedding — top 15 candidates
3 Keyword Matching
Extract keywords (stop-word filtered), ILIKE match against table/column names. Scores: exact=1.0, component=0.7, substring=0.5
4 FK Graph Expansion
From top 5 scored candidates, walk foreign key constraints to include necessary JOIN tables (relationship_score = 1.0)
5 Score & Prune
Weighted scoring formula, keep top 8 tables with all their columns
6 Resolve Glossary, Metrics, Knowledge, Dictionary
Match business terms + metric names in question text & by embedding similarity. Find top 5 knowledge chunks by vector similarity (keyword ILIKE fallback). Fetch value mappings for selected columns.
7 Find Similar Sample Queries
Top 3 validated sample queries by embedding similarity — used as few-shot examples
8 Assemble Prompt
Format into structured sections: Schema → Relationships → Glossary → Metrics → Knowledge → Dictionary → Examples → Constraints

Relevance Scoring Formula

score = 0.5 × embedding + 0.3 × keyword + 0.2 × fk_proximity + glossary_boost

Sample Queries

Validated NL→SQL pairs used as few-shot examples.

"total ECL by stage" → SELECT stage, SUM(ecl_lifetime) FROM ... GROUP BY stage

LLM Orchestration Layer

A multi-agent system with complexity-based model routing and an error recovery loop. All agents share the same provider-agnostic interface through BaseLLMProvider.

Complexity Router

TierSignal KeywordsModelMax Tokens
Simple what is, show me, single entity lookups claude-haiku-4-5-20251001 2,048
Moderate group by, join, total, average, per, by claude-sonnet-4-20250514 4,096
Complex trend, compare, window, YoY, cohort, percentile, pivot claude-sonnet-4-20250514 4,096

Four Specialized Agents

QueryComposerAgent

Input: NL question + assembled context

Output: SQL + explanation + confidence (0–1) + tables_used + assumptions

Response: JSON parse with regex fallback for code-fenced SQL

SQLValidatorAgent

Level 1: Static regex blocklist (DROP, DELETE, INSERT, ALTER, CREATE, TRUNCATE, pg_sleep, etc.)

Level 2: Schema reference check via sqlparse (tables/columns exist)

Status: VALID | UNSAFE | SYNTAX_ERROR | SCHEMA_MISMATCH

ResultInterpreterAgent

Input: Question + SQL + results preview (max 20 rows)

Output: NL summary + key highlights + suggested follow-up questions

Style: Data analyst — concise, insight-focused

ErrorHandlerAgent

Trigger: Validation failure OR database execution error

Input: Failed SQL + error + schema context + previous attempts

Output: Corrected SQL + explanation + should_retry flag

Max retries: 3 per phase (validation + execution = up to 6 total)

Provider Interface

class BaseLLMProvider(ABC):

Connector Layer

A plugin system for database connectivity. All connectors implement BaseConnector ABC and are registered in a global registry with connection pooling and health checks.

Connector Implementation Details

PostgreSQL

  • Driver: asyncpg (native async)
  • Read-only: conn.transaction(readonly=True)
  • Timeout: SET LOCAL statement_timeout
  • Introspection: information_schema + pg_catalog
  • Row limit: Auto LIMIT N+1 wrapping

BigQuery

  • SDK: google-cloud-bigquery (sync, wrapped in asyncio.to_thread)
  • Auth: Service account JSON (encrypted at rest)
  • Read-only: SQL blocklist (EXPORT DATA, LOAD DATA)
  • Introspection: INFORMATION_SCHEMA.TABLES/COLUMNS
  • Quoting: Backtick identifiers

Databricks

  • SDK: databricks-sql-connector (sync, wrapped in asyncio.to_thread)
  • Auth: Personal Access Token (PAT)
  • Read-only: SQL blocklist (COPY INTO, OPTIMIZE, VACUUM)
  • Introspection: Unity Catalog (information_schema) with Hive metastore fallback (SHOW/DESCRIBE)
  • Namespace: 3-level (catalog.schema.table)

Query Flow Pipeline

End-to-end flow when a user submits a natural language question through POST /api/v1/query.

1 User asks a question
"What is the total ECL by stage?"
2 Context Builder
Embed question → hybrid table search (vector + keyword + FK) → resolve glossary/metrics/knowledge/dictionary → find sample queries → assemble structured prompt
3 LLM Router
Estimate complexity (simple/moderate/complex) → select model (Haiku or Sonnet) → set temperature=0.0
4 QueryComposerAgent
LLM generates SQL from context + question. Parses JSON response (or fallback regex extraction). Returns SQL + explanation + confidence.
5 SQLValidatorAgent
Static blocklist check (DDL/DML/injection) + schema reference validation (sqlparse). If INVALID → ErrorHandlerAgent retries up to 3x.
Retry loop (max 3) — ErrorHandlerAgent corrects SQL
6 Execute via Connector
Read-only transaction + statement timeout + auto LIMIT wrapping. If DB error → ErrorHandlerAgent retries up to 3x.
Retry loop (max 3) — ErrorHandlerAgent corrects SQL on DB errors
7 ResultInterpreterAgent
LLM generates natural language summary, key highlights, and suggested follow-up questions from results (max 20 rows preview).
8 Save to History
QueryExecution record: question, generated_sql, final_sql, status, timing, tokens, retry_count, summary, is_favorite.

Security Architecture

Defense-in-depth strategy with multiple independent safety layers, ensuring no single bypass can allow destructive operations.

🔒

Read-Only Enforcement

PostgreSQL: asyncpg transaction(readonly=True). BigQuery & Databricks: read-only enforced via SQL blocklist (no native read-only mode).

🚫

SQL Blocklist

Static regex patterns block DDL, DML, admin commands, injection patterns (pg_sleep, dblink), BigQuery ops (EXPORT DATA, LOAD DATA), and Databricks ops (COPY INTO, OPTIMIZE, VACUUM).

🔑

Encrypted Credentials

Connection strings encrypted at rest with Fernet (AES-128-CBC). Never returned in API responses — only has_connection_string: boolean.

⏱

Statement Timeout

PostgreSQL: SET LOCAL statement_timeout. BigQuery & Databricks: asyncio.wait_for(). Default 30s, configurable per connection.

📊

Row Limits

Auto-wrapping queries with LIMIT N+1 to cap result size. Default 1,000 rows, configurable per connection. Truncation flag in response.

🌐

CORS Restriction

FastAPI CORS middleware restricts origins to configured list. Default: http://localhost:5173 only.
